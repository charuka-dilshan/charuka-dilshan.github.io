Detailed Course Notes

1. File Processing vs. Database Systems

Understanding the limits of old systems helps explain the need for databases.

File Processing Systems

Stores and manages data in one or more separate files. [cite_start]This was common on mainframe systems[cite: 2655, 2663].

Potential Problems:

    [cite_start]
  • Data Redundancy: Data is stored in several places, wasting space and making updates expensive [cite: 2676-2677].
    • [cite_start]
  • Data Integrity: The validity of data is compromised (e.g., a customer's address is updated in one file but not another)[cite: 2678].
    • [cite_start]
  • Rigid Data Structure: Files are inflexible and tied to the application [cite: 2679-2680].

Database Systems

[cite_start]

A collection of linked data files (tables) that form an overall data structure[cite: 2656, 2728]. [cite_start]A database avoids redundancy and supports a dynamic environment [cite: 2729-2730].

Advantages:

    [cite_start]
  • Scalability: Can be easily expanded or modified[cite: 2770].
    • [cite_start]
  • Controlled Redundancy: Data is stored in one place, minimizing duplication[cite: 2788].
    • [cite_start]
  • Data Independence: Systems are separate from how the data is physically stored[cite: 2791].
    • [cite_start]
  • Better Security: A Database Administrator (DBA) can define authorization procedures[cite: 2789].
    • [cite_start]
  • Flexible Data Sharing: Allows many users to view the same information in different ways[cite: 2780].

2. The Three-Schema Architecture

[cite_start]

Databases are described at three abstract levels [cite: 1970-1972]:

Level (Schema) Description
External (View) The highest level. Describes the portion of the database a specific user or application sees. [cite_start]There can be many external views [cite: 1994-1995].
Conceptual (Logical) Describes the entire database: all entities, attributes, and relationships. [cite_start]It is the community view, and there is only one [cite: 1997-1998].
Internal (Physical) The lowest level. [cite_start]Describes *how* the data is physically stored (data structures, access methods) [cite: 2001-2002].

3. Data Independence

[cite_start]

The main goal of the three-schema architecture is to separate user applications from the physical database[cite: 1977].

    [cite_start]
  • Logical Data Independence: The capacity to change the conceptual schema without having to change the external schema (user views)[cite: 2005].
    • [cite_start]
  • Physical Data Independence: The capacity to change the internal schema (e.g., adding an index to improve performance) without having to change the conceptual schema[cite: 2006].

4. Data Models

[cite_start]

A data model is a set of tools used to describe the structure of a database[cite: 2302].

  • High-Level (Conceptual): Uses concepts like entities, attributes, and relationships. (e.g., ER Model, Object-Oriented Model) [cite_start][cite: 2325, 2334, 2355].
  • Representational (Implementation): Used in commercial DBMS. (e.g., Relational Model, Hierarchical Model, Network Model) [cite_start][cite: 2327-2329, 2395].
    • [cite_start]
  • Low-Level (Physical): Describes the details of how data is stored (e.g., record structures, indexing)[cite: 2330].
[cite_start]

The Relational Model is the most common, viewing data as two-dimensional tables called relations[cite: 391, 401, 2396].

5. Database Management System (DBMS)

[cite_start]

A DBMS is a collection of tools, features, and interfaces to add, update, manage, access, and analyze a database[cite: 476, 2764].

[cite_start]

Five Software Components of a DBMS[cite: 483]:

  1. DBMS Engine: Accepts logical requests, converts them to their physical equivalent, and accesses the data. [cite_start]It separates the logical and physical views [cite: 514-515].
    2. [cite_start]
  2. Data Definition Subsystem: Helps create and maintain the **Data Dictionary** and define the file structure [cite: 487, 525-526].
  3. Data Manipulation Subsystem: Helps you add, change, and delete information. [cite_start]This is the primary user interface and includes tools like SQL and Query-by-Example (QBE) [cite: 489, 532-533, 536-537].
    4. [cite_start]
  4. Application Generation Subsystem: Contains facilities to help developers create transaction-intensive applications (e.g., data entry forms)[cite: 491, 541, 543].
  5. Data Administration Subsystem: Helps manage the overall database environment. This includes:
      [cite_start]
    • Backup and Recovery [cite: 550, 558]
      • [cite_start]
    • Security Management [cite: 551, 561]
      • [cite_start]
    • Concurrency Control (what happens if two people make changes at the same time) [cite: 553, 569-570].

6. Data Warehouses and Data Mining

Data Warehouse

    [cite_start]
  • A **Data Warehouse** is a logical collection of information, gathered from operational databases, used to create business intelligence that supports decision-making [cite: 590-594].
    • [cite_start]
  • Data warehouses support **OLAP** (Online Analytical Processing) for decision support[cite: 582].
    • [cite_start]
  • They do **NOT** support **OLTP** (Online Transaction Processing), which is handled by operational databases[cite: 583].
    • [cite_start]
  • They are **multidimensional**, often called "hypercubes," allowing users to "slice and dice" data (e.g., see sales by product, by territory, by year)[cite: 620, 623].

Data Mining

[cite_start]

Data-Mining Tools are the software tools you use to query and analyze the information in a data warehouse[cite: 584, 627]. Examples include: [cite_start]

  • Query-and-Reporting Tools [cite: 628]
    • [cite_start]
  • Intelligence Agents [cite: 629]
    • [cite_start]
  • Statistical Tools [cite: 631]
